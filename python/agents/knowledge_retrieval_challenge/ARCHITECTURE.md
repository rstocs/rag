# System Architecture & Design Document

This document outlines the design decisions, model selection justifications, and architectural patterns used to build the production-grade SpaceX Regulatory RAG pipeline.

## 1. Architectural Overview

The goal of this system is to ingest a highly complex, 483-page regulatory application (containing dense tables, sparse spatial maps, laboratory footnotes, and mixed languages) and answer arbitrary queries with 1.00 Faithfulness (zero hallucinations). 

To achieve this without relying on heavy abstractions like LangChain or LlamaIndex, the architecture is broken down into three decoupled subsystems:
1. **Intelligent Ingestion Pipeline:** A multi-modal router and dual-indexing system.
2. **Hybrid Retrieval System:** A multi-stage retrieval pipeline using BM25, Vector Embeddings, and a Cross-Encoder.
3. **Grounded Generation Engine:** A strict, prompt-engineered generator designed to fail gracefully on unanswerable queries.

---

## 2. Model Selection & Justification

The system relies on the **Google GenAI SDK (Vertex AI)** using the Gemini family of models. Selecting the right model for the right task was critical to balancing cost, token limits, and reasoning capability.

### Ingestion & Extraction Models
*   **Intelligent Router (`gemini-2.5-flash`)**: Used to rapidly classify pages into `TABLE_DATA`, `ADMIN_FORM`, `VISUAL_DIAGRAM`, or `STANDARD_TEXT`. We chose **Gemini 2.5 Flash** because routing requires minimal reasoning but high speed and low cost across hundreds of pages.
*   **Visual Extractor (`gemini-2.5-pro`)**: Used to extract complex layouts (dense tables, spatial maps, and forms). We chose **Gemini 2.5 Pro** because extracting nested tables, understanding laboratory footnotes, and calculating spatial relationships on dense architectural diagrams requires the highest tier of visual-spatial reasoning. Flash models struggled to accurately reconstruct multi-page tables.
*   **Text Formatter (`gemini-2.5-flash`)**: Used to clean up raw OCR text for standard paragraph pages.
*   **Embedding Model (`text-embedding-004`)**: Used to embed all textual chunks. We selected `text-embedding-004` because of its high dimensional capacity and native support for semantic regulatory text.

### Generation & Query Models
*   **Synthetic Data Generator (`gemini-2.5-pro`)**: Used at the end of ingestion to generate a global document summary and synthetic Q&A pairs (HyDE). Pro was chosen to handle the massive context window (summarizing the entire document) without losing edge details.
*   **Query Expander (`gemini-2.5-flash`)**: Used to expand the user's query into synonyms and bilingual terms (e.g., Spanish translations). Flash is perfectly capable of fast keyword generation.
*   **Final Answer Generator (`gemini-2.5-pro`)**: Used to synthesize the final answer from the retrieved context. Pro was chosen because following strict negative constraints (e.g., "Do not perform undocumented mathematical operations", "Only answer if the context contains the answer") requires advanced instruction-following capabilities to prevent hallucinations.

## 3. Ingestion Pipeline & Dual-Indexing

Standard naive text chunking fails on regulatory PDFs because structural context (like table headers or page numbers) is lost. We solved this by implementing a **Dual-Indexing Strategy**.

### Intelligent Modality Routing
Processing a 483-page PDF through a high-tier Vision model is extremely slow and expensive. We built a router that reads the raw PyMuPDF text of a page and decides its modality:
1.  **STANDARD_TEXT:** Fast-pathed to a cheap text formatter.
2.  **TABLE_DATA & ADMIN_FORM:** Routed to the Vision model to reconstruct tabular relationships into readable semantic paragraphs.
3.  **VISUAL_DIAGRAM:** Routed to the Vision model specifically to describe spatial relationships between labeled elements on the map.

### Dual-Indexing
During early testing, we discovered that while the Vision model perfectly extracted table data, it occasionally dropped "boilerplate" text (like the name of the laboratory in the header). 
To solve this, we **dual-index** every page:
1.  **VLM Extraction:** The semantic summary generated by the Vision model.
2.  **Raw OCR:** The raw string extracted by PyMuPDF.

Both are embedded and stored. This guarantees that complex structural relationships (captured by the VLM) and exact raw strings (captured by OCR) are both available to the retriever.

## 4. Retrieval Architecture

To achieve 1.00 Relevance, the retrieval system uses a **multi-stage pipeline**:

1.  **Query Expansion:** The user's query is expanded into synonyms, related technical terms, and Spanish translations (since the document contains bilingual public notices).
2.  **Hybrid Retrieval (Top 100):**
    *   **Dense Retrieval (FAISS):** Semantic matching using `text-embedding-004`.
    *   **Sparse Retrieval (BM25):** Exact keyword matching using a custom BM25 index (crucial for exact IDs like "Outfall 001" or chemical acronyms like "VOC").
3.  **Cross-Encoder Re-ranking (Top 10):** The top 100 chunks from the hybrid search are passed through a HuggingFace Cross-Encoder (`cross-encoder/ms-marco-MiniLM-L-6-v2`). The cross-encoder evaluates the exact semantic relationship between the query and each chunk, re-ordering them to bubble the absolute most relevant chunks to the top.

## 5. Guardrails & Generation

In regulatory and compliance environments, a hallucination is a critical failure. The system is hardened to prioritize **Faithfulness** above all else.

*   **Zero-Temperature Determinism:** All generation LLM calls operate at `temperature=0.0`.
*   **Numerical Guardrails:** The generator is explicitly prompted *not* to perform undocumented mathematical operations (e.g., converting µg/L to mg/L if the context does not explicitly provide the conversion). It is instructed to state that the data is only available in its original units.
*   **Strict Abstention:** If the retrieved context does not contain the answer (e.g., "What is SpaceX's stock price?"), the generator is instructed to explicitly state: "I cannot answer this based on the context."

---

## 6. Production Readiness

*   **Concurrency & Rate Limiting:** Built-in `tenacity` exponential backoff and `asyncio.Semaphore` concurrency limits (tuned to 5 concurrent Vertex AI requests) prevent `429 RESOURCE_EXHAUSTED` quota limits.
*   **Context Window Management:** The synthetic summarization phase aggressively truncates input text to 500,000 characters to ensure it never exceeds the API context limits.
*   **Deterministic Evaluation:** The system includes a full LLM-as-a-judge evaluation suite (`evaluator.py`) that scores Faithfulness, Relevance, and Context Relevance across 23 complex edge-case questions.
